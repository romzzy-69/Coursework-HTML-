1. Profile Page

View Profile Page | View Validation

HTML Structure

I used a simple structure for the profile page. The main layout has two columns: the left side shows the steps, and the right side displays the user's profile. I used the shared template from Student 1, so the header, navigation and the footer are consistent with the rest of the website.

Key HTML elements I used:

  • <header> — for the navigation bar (shared template)
  • <main> — contains the main content of the page
  • <section> — groups the profile layout together
  • <div> — for layout containers and step items
  • <button> — for user interactions (Start and Skip)
  • <footer> — shared footer with links

CSS Techniques

I used Flexbox to arrange the two columns side by side. The display: flex property makes the children sit in a row, and gap: 30px adds spacing between the columns. This was a simple way to make the layout responsive — when the screen gets smaller (@media (max-width: 900px)), the columns stack on top of each other using flex-direction: column.

I also used CSS classes like .step-complete and .step-skipped to change the appearance of steps based on their status, which helps users see their progress. For the progress bar, I used a simple <div> inside another <div> and changed the width using JavaScript. The colours match the website's colour palette (blue and green).

JavaScript Implementation

The profile builder uses the prompt() function to collect data, just like we learned in the lectures. I created a profile object to store all the user's answers. Each step has 3 prompts, and there are 3 steps in total.

The main functions I wrote are:

  • startStep(stepNum) — runs the prompts for a step and stores answers. Each answer is checked with if (name !== null && name.trim() !== "") to handle cases where the user cancels or enters only spaces.
  • skipStep(stepNum) — skips a whole step without asking prompts. All fields in that step are set to empty strings, so they won't appear in the profile.
  • updateDisplay() — updates the profile card on the right side. It checks each field and only shows the ones that have been answered. I used variables like fullNameField, userNameField and actionField to store the DOM elements.
  • updateProgress() — updates the progress bar and step counter. It counts how many fields have answers and calculates the percentage.
  • enableNext(stepNum) — unlocks the next step's buttons. It finds the buttons using document.getElementById("startBtn" + next) and document.getElementById("skipBtn" + next).

When the user skips a question or a whole step, the data is set to an empty string. This means it won't appear in the profile display and the progress bar won't count it as answered. The user can always come back to skipped steps later. The buttons for each step are always enabled once the step has been unlocked.

I used DOM manipulation (document.getElementById() and innerHTML) to update the profile display without refreshing the page. This was covered in the lecture notes about JavaScript and DOM.

Variable naming: I used descriptive variable names like startBtn1, skipBtn1, fullNameField, userNameField, actionField, and personalGoalField to make the code easier to understand.

2. Sitemap Page

View Sitemap Page | View Validation

HTML Structure

The sitemap is built using SVG (Scalable Vector Graphics), which we learned about in the lecture notes. I used the shared template for the header and footer.

Key HTML elements I used:

  • <svg> — the main container for the sitemap diagram
  • <g> — groups elements together
  • <rect> — creates the boxes for each page
  • <text> — adds the page names inside the boxes
  • <path> — draws the lines connecting pages
  • <a> — makes each box clickable and links to the page
  • <title> — provides accessible names for screen readers

SVG Design

I designed the sitemap to show the website hierarchy. The Splash Screen is at the top, leading to the Home page. From Home, there are branches to all the main pages. The Content Pages group then branches out to the four individual content pages.

I used different colours for different levels:

  • Dark blue (#003b73) — Entry pages (Splash, Home)
  • Light blue (#00a6c8) — Main navigation pages
  • Green (#2a9d8f) — Content group
  • White with green border — Individual content articles

Interactivity

Each node in the sitemap is a clickable link that takes the user to the corresponding page. I wrapped each SVG node in an <a> tag with the href attribute pointing to the correct page.

I added hover effects using CSS. When you hover over a box, it changes to coral red (#ff6b6b) and the text turns white. This was done using the :hover selector in CSS:

                    .sitemap-svg .node:hover rect {
                    fill: #ff6b6b;
                    stroke: #ff6b6b;
                    stroke-width: 3;}

                    .sitemap-svg .node:hover text {
                    fill: white;
                    }

For accessibility, I added tabindex="0" to each link so users can navigate using the keyboard. I also added <title> elements inside each link to provide tooltips and information for screen readers. The visible text on each node already shows the page name, so users can easily see what each link does.

I used the viewBox attribute on the SVG to make it responsive. The viewBox defines the coordinate system of the SVG and combined with width: 100% and height: auto in CSS, the sitemap scales properly on different screen sizes. I also added min-width: 700px to prevent the text from becoming too small on very narrow screens.

4. Challenges and Lessons Learned

Challenge 1: Making the SVG Sitemap Accessible

When I first created the SVG sitemap, I didn't think about accessibility. I realised that screen readers wouldn't be able to read the text inside the SVG and keyboard users couldn't navigate through the links.

How I fixed it: I added tabindex="0" to each link so they could be tabbed through. I also added <title> elements to describe each link. For keyboard focus, I used the :focus-visible CSS selector to show a visible outline when a link is focused.

What I learned: Accessibility is important and should be considered from the start, not as an afterthought. Small changes can make a big difference for users with disabilities.

Challenge 2: Managing Profile Data with JavaScript

Storing the user's answers and updating the display was tricky at first. I had to think about how to handle skipped questions and make sure the progress bar updated correctly.

How I fixed it: I created a profile object to store all the data. Each field starts as an empty string. When a question is answered, the value is stored. When a question is skipped, the value stays empty. The updateDisplay() function checks each field and only shows the ones with data.

What I learned: Planning the data structure before writing the code makes it much easier. Using a simple object to store data and separate functions to update the display helped keep the code organised.

Challenge 3: Validating HTML

When I first validated my pages, I got errors about heading structure (skipping heading levels) and a section without a heading. I didn't realise that heading order mattered for accessibility.

How I fixed it: I fixed the heading structure so it goes h1 → h2 → h3 in order. I also changed a section without a heading to a <div> instead.

What I learned: Valid HTML is important for accessibility and SEO. I should validate my pages regularly as I work, not just at the end.

5. Accessibility Considerations

I made sure to follow accessibility principles on all my pages:

  • Semantic HTML: I used proper heading levels (h1, h2, h3) and semantic elements like <header>, <main>, <section>, <aside>, and <footer>.
  • Alt Text: All images have descriptive alt text that explains what the image shows.
  • Keyboard Navigation: The SVG sitemap can be navigated using the Tab key, with a visible focus state.
  • Readable Text: I used good contrast between text and backgrounds (dark blue on white, white on dark blue).
  • Focus Indicators: I used the :focus-visible selector to show a visible outline on focused elements.

These choices make the website more usable for everyone, including people who use screen readers or who can't use a mouse.
