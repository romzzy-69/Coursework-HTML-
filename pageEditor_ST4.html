Profile Page

Technical Description

HTML Structure: I used a simple structure for the profile page.
The main layout has two columns:
the left side shows the steps and the right side displays the user's profile. I used the shared template from Student 1, so the header, navigation and the footer are consistent with the rest of the website.

Key HTML elements I used:
<header>, <main>, <section>, <div>, <button> and <footer>.

CSS Techniques:
I used Flexbox to arrange the two columns side by side. The "display: flex" property makes the children sit in a row and "gap: 30px" adds spacing between the columns. This was a simple way to make the layout responsive.
When the screen gets smaller (@media (max-width: 900px)), the columns stack on top of each other using "flex-direction: column". I also used CSS classes like ".step-complete" and ".step-skipped" to change the appearance of steps based on their status, which helps users see their progress.
For the progress bar, I used a simple "<div>" inside another "<div>" and changed the width using JavaScript. The colours match the website's colour palette.

JavaScript Implementation:
The profile builder uses the "prompt()" function to collect data, just like we learned in the lectures.
I created a "profile" object to store all the user's answers. Each step has 3 prompts and there are 3 steps in total.

The main functions I wrote are:

  • startStep(stepNum) — runs the prompts for a step and stores answers. Each answer is checked with "if (name !== null && name.trim() !== "")" to handle cases where the user cancels or enters only spaces.

  • skipStep(stepNum) — skips a whole step without asking prompts. All fields in that step are set to empty strings, so they won't appear in the profile.

  • updateDisplay() — updates the profile card on the right side. It checks each field and only shows the ones that have been answered. I used variables like "fullNameField", "userNameField" and "actionField" to store the DOM elements.

  • updateProgress() — updates the progress bar and step counter. It counts how many fields have answers and calculates the percentage.

  • enableNext(stepNum) — unlocks the next step's buttons. It finds the buttons using "document.getElementById("startBtn" + next)" and "document.getElementById("skipBtn" + next)".

I used DOM manipulation ("document.getElementById()" and "innerHTML") to update the profile display without refreshing the page.
This was covered in the lecture notes about JavaScript and DOM.
I used descriptive variable names like "startBtn1", "skipBtn1", "fullNameField", "userNameField", "actionField", and "personalGoalField" to make the code easier to understand.

Accessibility

I ensured the profile page is accessible by using semantic HTML elements and proper heading levels ("h1, h2, h3").
All buttons have clear labels and the progress bar includes a live region (aria-live="polite") to announce updates to screen reader users. I also added a visible focus state for interactive elements.

Sitemap Page

Technical Description

HTML Structure: The sitemap is built using SVG (Scalable Vector Graphics), which we learned about in the lecture notes. I used the shared template for the header and footer.

Key HTML elements I used:
"<svg>", "<g>", "<rect>", "<text>", "<path>", "<a>" and "<title>".

SVG Design:
I designed the sitemap to show the website hierarchy. The Splash Screen is at the top, leading to the Home page. From Home, there are branches to all the main pages. The Content Pages group then branches out to the four individual content pages.

I used different colours for different levels:

  • Dark blue (#003b73) — Entry pages (Splash, Home)
  • Light blue (#00a6c8) — Main navigation pages
  • Green (#2a9d8f) — Content group
  • White with green border — Individual content articles

Interactivity:
Each node in the sitemap is a clickable link that takes the user to the corresponding page. I wrapped each SVG node in an <a> tag with the href attribute pointing to the correct page. I added hover effects using CSS. When you hover over a box, it changes to coral red (#ff6b6b) and the text turns white.
This was done using the :hover selector in CSS.

I used the viewBox attribute on the SVG to make it responsive. The viewBox defines the coordinate system of the SVG and combined with width: 100% and height: auto in CSS, the sitemap scales properly on different screen sizes.
I also added min-width: 700px to prevent the text from becoming too small on very narrow screens.

Accessibility

For accessibility, I added tabindex="0" to each link so users can navigate using the keyboard. I also added <title> elements inside each link to provide tooltips and information for screen readers. The visible text on each node already shows the page name, so users can easily see what each link does. I also ensured a visible focus state using the :focus-visible CSS selector.

Content Page

Technical Description

HTML Structure: My content page is about "Sustainable Fishing & Marine Conservation" which relates to SDG 14 (Life Below Water). I used the shared template for the header and footer and used the .content-layout class from the global CSS for the two-column layout (sidebar + main content).

Key HTML elements I used:
"<aside>", "<section>", "<h2>", "<img>", "<a href="#section">" and "<a href="#top">".

Internal Navigation:
I added a sidebar on the left side of the page with links to each section. When you click a link, the page scrolls to that section. I used anchor links like #section-1, #section-2, etc. and added scroll-margin-top in CSS so the heading doesn't hide behind the sticky header.
I also added a "Go to Top" button that stays in the bottom-right corner of the screen using position: fixed. This was covered in the lecture notes about CSS positioning.

Images and Alt Text:
I included 6 images on my content page, each with descriptive alt text. All images use the .article-image class from the global CSS, which adds rounded corners and a shadow.

Accessibility

I used proper heading levels to structure the content and included descriptive alt text for all images.
The sidebar provides clear navigation for users and the "Go to Top" button is easily accessible. I also ensured good colour contrast between text and backgrounds throughout the page.

Challenges and Lessons Learned

Challenge 1: Making the SVG Sitemap Accessible

When I first created the SVG sitemap, I didn't think about accessibility. I realised that screen readers wouldn't be able to read the text inside the SVG and keyboard users couldn't navigate through the links.

How I fixed it:
I added tabindex="0" to each link so they could be tabbed through. I also added <title> elements to describe each link.
For keyboard focus, I used the :focus-visible CSS selector to show a visible outline when a link is focused.

What I learned:
Accessibility is important and should be considered from the start, not as an afterthought. Small changes can make a big difference for users with disabilities.

Challenge 2: Managing Profile Data with JavaScript

Storing the user's answers and updating the display was tricky at first. I had to think about how to handle skipped questions and make sure the progress bar updated correctly.

How I fixed it:
I created a profile object to store all the data. Each field starts as an empty string. When a question is answered, the value is stored. When a question is skipped, the value stays empty. The updateDisplay() function checks each field and only shows the ones with data.

What I learned:
Planning the data structure before writing the code makes it much easier.
Using a simple object to store data and separate functions to update the display helped keep the code organised.

Challenge 3: Validating HTML

When I first validated my pages, I got errors about heading structure (skipping heading levels) and a section without a heading.
I didn't realise that heading order mattered for accessibility.

How I fixed it:
I fixed the heading structure so it goes h1 → h2 → h3 in order. I also changed a section without a heading to a <div> instead.

What I learned:
Valid HTML is important for accessibility and SEO. I should validate my pages regularly as I work, not just at the end.
