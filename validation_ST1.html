Reflection on Validation

Using the W3C Validator was an eye-opening experience because it caught outdated habits and structural issues that I couldn't see just by looking at the visual layout. For instance, it flagged numerous messages across my pages for using trailing slashes (/>) on void elements like <img>, <br>, and <meta> tags. While this used to be standard practice, the validator taught me that modern HTML5 no longer requires them, prompting me to clean up my syntax globally.

It also caught a structural warning on my Content Page where my <article> element lacked a direct identifying heading. Even though my layout looked perfect and my sections had nested <h2> tags, the validator reminded me that strict semantic HTML requires clear document outlines. I fixed this by adding a visually hidden <h2> to satisfy the accessibility outline without breaking my CSS design.

Ultimately, this process proved that just because a page renders correctly in a browser doesn't mean the code is structurally sound under the hood. Validating the pages forced me to write cleaner, more professional HTML that is accessible to screen readers and fully compliant with modern web standards.
