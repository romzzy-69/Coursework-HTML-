Feedback & Program Directory

Technical Description

HTML Structure: This page uses the shared template for the header, navigation, and footer. The main content is split into two areas using a <section class="feedback-layout">: a scrollable <div class="directory-panel"> on the left containing the Opportunity Directory, and a <div class="form-panel"> on the right containing the feedback form.

Each opportunity is marked up as an <article class="program-card">, and I added a custom data-category attribute to every card (e.g. data-category="nearest") so the JavaScript filter can target the correct cards without relying on their visible text.

The feedback form is grouped using <fieldset> and <legend> elements ("Personal Details" and "Your Feedback") as required by the coursework brief. I used a mix of input types — type="email", type="tel", type="date", and type="datetime-local" — along with a <textarea maxlength="300"> for the comment field, and a <select> for the topic dropdown.

Below the form, a <section class="reviews-section"> displays a .rating-summary card alongside three .review-card elements, each showing a heading, star rating, short review, and reviewer meta information.

CSS Techniques:
The two-column layout of the directory and form is built with CSS Grid (display: grid; grid-template-columns: 1.1fr 1fr;), which collapses to a single column on smaller screens using a media query. Inside the directory, I used Flexbox (display: flex; flex-direction: column;) on .directory-scroll to stack the cards vertically inside a fixed-height, scrollable container.

One issue I ran into was that hiding filtered-out cards with the hidden attribute didn't visually work at first, because my .program-card { display: flex; } rule had the same CSS specificity as the browser's built-in [hidden] { display: none; } rule, and mine was loading later in the cascade. I fixed this by adding a more specific rule, .program-card[hidden] { display: none; }, which reliably overrides the flex display whenever a card is hidden.

I reused the shared :root colour variables across the page for consistency, and used box-shadow and border-radius variables to keep the cards, form panel, and rating summary visually unified.

JavaScript Implementation:
All interactivity on this page is JavaScript-driven, in line with the coursework requirement that hover-only CSS effects are not sufficient. My script is wrapped in a single DOMContentLoaded listener and is split into four parts:

  • Live character counter — an input event listener on the message <textarea> calls updateCharCount(), which updates the charCount span on every keystroke and turns the counter red once the 300-character limit is reached.
  • Field validation — I built a validators object containing a small function for each required field (fullName, email, telephone, topic, message), each returning an error string or an empty string if the value is valid. A shared showError() function then adds or removes an invalid CSS class on the field and writes the message into its matching <span class="error-msg">. I looped over Object.keys(validators) with .forEach() to attach the correct event (change for the topic <select>, input for everything else) without repeating code for each field individually.
  • Form submission — on submit, I call e.preventDefault(), re-run every validator, and either show a red "please fix the highlighted fields" message or a green "Thank you for your feedback!" confirmation inside a <p role="status" aria-live="polite">, then reset the form.
  • Opportunity Directory filter — a change listener on the sortDropdown calls filterPrograms(), which compares the dropdown's selected value against each card's dataset.category and toggles the hidden attribute accordingly, also showing a "No programs match this filter yet" message if the result set is empty.

Accessibility

Every form field has an associated <label>, and the two field groups use <fieldset>/<legend> so screen readers announce the section a field belongs to. The dropdown filter has a visually hidden <label> for the same reason. The confirmation message uses role="status" and aria-live="polite" so screen reader users are notified of a successful or failed submission without needing to refocus the page. All programme images include descriptive alt text, and the scrollable directory container has a visible :focus-visible outline for keyboard users.

Team Page

Technical Description

HTML Structure: The page uses the shared header, navigation, and footer. The main content is a <section class="team-grid"> containing four <article class="team-card" tabindex="0"> elements, one per team member. Each card contains a circular photo (<img class="team-photo">), the member's name in an <h2>, their role in a <p class="team-role">, a short hint prompting the user to hover, and a <div class="team-details"> holding their full contribution text, which stays hidden until the card is interacted with.

CSS Techniques:
The four cards are laid out using CSS Grid (display: grid; grid-template-columns: repeat(4, 1fr);), with media queries reducing this to two columns and then one column as the screen narrows. Each photo is made circular using border-radius combined with object-fit: cover, so real photos of differing proportions are cropped neatly instead of being stretched.

The expand-on-hover behaviour was intentionally built without display: none/block toggling. Instead, .team-details starts at max-height: 0; overflow: hidden; and transitions to max-height: 400px; on :hover and :focus-within. Because the element still takes up real space in the document flow rather than being positioned absolutely, animating its height physically pushes the rest of the page — including the footer — downward as it opens, rather than overlaying on top of the content.

I combined the :hover selector with :focus-within on every interactive rule (card lift, photo scale, and details reveal) so that the exact same visual effect is triggered whether a user hovers with a mouse or reaches the card using the keyboard.

JavaScript Implementation:
This page does not use JavaScript. Per the coursework specification, the Team page's interactivity is permitted to be CSS-based (unlike the Gallery or Feedback pages), so I implemented the hover/focus reveal entirely with the :hover and :focus-within pseudo-classes and CSS transitions described above.

Accessibility

Every .team-card has tabindex="0", making it reachable by keyboard Tab navigation even though it is not a naturally focusable element like a link or button. Pairing :focus-within with :hover throughout the CSS ensures keyboard users see identical contribution details to mouse users, rather than being locked out of the hover-only content. Every team photo includes descriptive alt text (e.g. "Student 1 photo"), and the card's visible focus outline and lift animation give a clear indicator of which card is currently active.

Content Page

Technical Description

HTML Structure: The Content Page follows the shared website template by reusing the common header, navigation bar and footer. The page begins with a hero section (<section class="content-hero">) introducing the topic Marine Biodiversity & Endangered Species. The main content is organised using a <div class="content-layout">, which separates the page into a sidebar navigation and the main article.

The sidebar is contained inside an <aside class="article-sidebar"> and includes an ordered list of internal navigation links. Each navigation link connects to a corresponding content section using matching href and id attributes. The main article is wrapped inside <article class="article-content"> and is divided into six semantic <section> elements covering marine biodiversity, coral reefs, sea turtles, dolphins and whales, the importance of biodiversity, and methods of protecting endangered marine species.

Each section follows a consistent structure consisting of a heading, descriptive text, an informative callout box, an image, and an image caption. This consistent structure improves readability and creates a uniform appearance across the page.

CSS Techniques:
The overall page layout is created using CSS Grid with grid-template-columns: 300px 1fr;, producing a two-column layout that automatically changes into a single-column layout on smaller screens using media queries. The sidebar uses position: sticky, allowing the navigation menu to remain visible while users scroll through the article.

Images are styled using width:100%, border-radius, and box-shadow to ensure they are responsive while maintaining a consistent appearance with the rest of the website. Informational callout boxes are highlighted using custom styling, including contrasting background colours, borders, spacing and padding to distinguish important facts from the main content.

To improve internal navigation, each content section uses the CSS property scroll-margin-top. This prevents the fixed website header from covering section headings when users click links in the sidebar navigation.

Navigation Implementation:
The internal navigation is implemented entirely using HTML anchor links rather than JavaScript. Every sidebar link references the unique id of its corresponding section, allowing users to jump directly to specific topics. A fixed "Back to Top" button positioned at the bottom-right corner links to the element with id="top", allowing users to quickly return to the beginning of the article.

Accessibility

The page makes extensive use of semantic HTML elements including <main>, <article>, <aside>, and <section> to improve navigation for screen reader users. Every image contains descriptive alternative text, the heading hierarchy follows a logical structure, and internal navigation links can be accessed entirely using the keyboard. The Back to Top button also provides an additional navigation aid for users reading long pages.

Challenges and Lessons Learned

Developing the Feedback, Team, and Content pages significantly improved my understanding of semantic HTML, responsive CSS layouts, accessibility, and JavaScript validation techniques.

One of the biggest challenges I encountered was implementing the Opportunity Directory filter on the Feedback page. Although the JavaScript correctly applied the hidden attribute to filtered program cards, the cards continued to appear because my .program-card { display: flex; } rule overrode the browser's default [hidden] { display: none; } rule. After researching CSS specificity and the cascade, I solved the issue by creating the more specific selector .program-card[hidden] { display: none; }, allowing the filter to work correctly.

Another challenge occurred while building my Content Page. Initially, several internal navigation links did not scroll to the correct sections because the href values in the sidebar did not exactly match the corresponding section id attributes. After correcting the IDs and adding scroll-margin-top to each section, the navigation worked correctly without the sticky header covering the headings.

I also encountered a styling issue where my content images were not displaying rounded corners even though the shared stylesheet already contained the required CSS. After comparing my HTML structure with the stylesheet and checking the applied selectors using browser developer tools, I corrected the HTML so the existing CSS rules were applied correctly.

Throughout this coursework I gained practical experience using CSS Grid to build responsive page layouts, Flexbox to align components, semantic HTML to improve accessibility, and JavaScript to implement form validation, live character counting, and dynamic filtering. I also became more confident using browser developer tools to inspect CSS rules, debug layout problems, and resolve conflicts between HTML structure and external stylesheets.
