Homepage

Technical Description

HTML Structure:As Student 1, I was responsible for the shared website template. I chose to use semantic HTML5 elements like <header>, <nav>, <main>, and <footer> because they provide clear meaning to the page structure, which improves both accessibility for screen readers and overall code readability.

On the Content Page, I used a <div class="content-layout"> to separate the page into an <aside> (for the sticky sidebar menu) and an <article> (for the main educational content). This logical separation ensures the user can easily navigate the text. I used heading tags (<h1> to <h2>) to establish a strict content hierarchy.

On the Gallery Page, I structured the grid using <div class="gallery-card"> containers. I utilized the <div class="gallery-card"> HTML attribute on these cards to seamlessly link the structural elements to the JavaScript data array without cluttering the HTML with hidden text.

CSS Techniques:
My CSS strategy focused on creating a responsive, cohesive visual identity using global variables (:root) for colors and spacing. I avoided copying large code sections by relying on reusable classes.
Layouts (Grid vs. Flexbox): I justified using CSS Grid (display: grid) for the mission cards and the gallery because it perfectly handles rigid, two-dimensional card layouts. I chose Flexbox (display: flex) for the navigation bar and alignment within containers because of its flexibility in distributing space along a single row.
Positioning: I applied position: sticky to the <header> and the article sidebar. The purpose of this was to keep vital navigation tools visible on the screen as the user reads long articles. I used position: fixed for the "Go to Top" button and the modal backdrop so they sit on top of the entire document.

JavaScript Implementation:
My JavaScript logic is focused on the Gallery Page to drive the interactivity requirements without relying on CSS-only hacks. Instead of hardcoding all the expanded descriptions into the HTML, I created a galleryData array to store objects containing the titles, descriptions, and SDG relevance for each image. This separates data from presentation.

The main event listeners and functions I wrote are:

  • Hover Event Listeners (mouseenter and mouseleave) — I used a .forEach() loop to attach these to every card. They dynamically add or remove the overlay-visible CSS class to show a dark blue overlay with a "View Details" button, ensuring the hover effect is JavaScript-driven.

  • Card click Listener — Handles the click-to-expand feature. It reads the card's index using parseInt(card.getAttribute('data-index')). It then pulls the correct data from galleryData and updates the DOM using textContent and setAttribute on variables like targetTitle, targetImage, and targetDesc. Finally, it adds the modal-open class to the backdrop to fade the modal into view.

  • updateDisplay() — updates the profile card on the right side. It checks each field and only shows the ones that have been answered. I used variables like "fullNameField", "userNameField" and "actionField" to store the DOM elements.

  • Dropdown change Listeners — Handles the user customization requirement. When a user selects a new theme or font, the listener reads e.target.value. It removes the existing theme/font classes from the modalBox using classList.remove() and applies the newly selected class using classList.add('theme-' + selectedTheme).

I used standard DOM manipulation (document.getElementById() and document.querySelectorAll()) to update the modal display in real-time without refreshing the page. I also used descriptive variable names like modalBox, targetTitle, themeSelect, and closeTrigger to make the code highly readable and easy to follow.

Accessibility

I ensured my pages are accessible by using semantic HTML elements and logical heading levels ("h1, h2, h3").
All images include descriptive alt text for screen reader users, and the global colour palette ensures high contrast readability. I also implemented a global visible focus state (:focus-visible) to fully support keyboard navigation.

Gallery Page

Technical Description

HTML Structure: The gallery relies on the shared template for consistent header and footer navigation. The main content area features a 6-card grid, and I built a hidden modal structure (<div class="modal-backdrop">) at the bottom of the page to handle the extended view.

Key HTML elements I used:
"<div>", "<img>", "<button>", "<select>", and "<label>". I heavily utilized custom data-index attributes on each card to seamlessly link the HTML elements to the JavaScript data array.

CSS Design & Techniques:
To create the main gallery, I used CSS Grid (display: grid) with grid-template-columns: repeat(auto-fit, minmax(300px, 1fr)). This was a highly efficient way to make a responsive layout that automatically reflows on smaller screens without needing complex media queries.

For the extended view, I used position: fixed to ensure the modal backdrop overlays the entire screen, locking the user's focus. The modal itself also uses a 2-column Grid for a clean, side-by-side layout (image on the left, text and controls on the right). I applied smooth CSS properties (transition: opacity 0.3s ease) so the modal fades in naturally rather than snapping abruptly.


Interactivity (JavaScript):
To meet the strict coursework requirements, I completely avoided CSS-only hover hacks. Instead, I used a .forEach() loop to attach mouseenter and mouseleave event listeners to every card, which dynamically adds or removes an overlay-visible CSS class to reveal the "View Details" button.

When a card is clicked, the script reads its data-index, pulls the corresponding information from the galleryData array, and injects it into the hidden modal using textContent and setAttribute().

To handle the user customization requirement, I attached change event listeners to the Theme and Font <select> dropdowns. When a user picks a new option, the script removes the old CSS classes from the modal's classList and applies the new ones (e.g., .theme-ocean or .font-mono), allowing the user to change the visual design in real-time without refreshing the page.

Accessibility

To ensure the gallery is highly accessible, every <img> tag includes a descriptive alt attribute so visually impaired users using screen readers understand the visual content of the SDG 14 images. Within the extended view modal, I paired each <select> dropdown with a clear, associated <label> tag to support form accessibility. Finally, I ensured that all interactive elements including the cards, close buttons, and dropdowns trigger the global :focus-visible CSS outline, providing a clear visual indicator for users navigating the site exclusively with a keyboard.

Content Page

Technical Description

HTML Structure: My content page focuses on "Coral Reefs & Coastal Ecosystems," directly relating to SDG 14. I utilized the shared template for the header and footer, and structured the main body using a two-column CSS Grid layout (.content-layout) to separate an <aside> sidebar from the main <article> content.

Key HTML elements I used:
"<aside>", "<article>", "<section>", "<h2>", "<img>", and "<div class="callout">".

Internal Navigation & CSS:
I built a sidebar menu containing an ordered list of anchor links (e.g., #what-are-corals). I applied position: sticky to this sidebar so it remains visible alongside the reader. To ensure the section headings aren't hidden behind the global sticky header when a link is clicked, I applied scroll-margin-top: 120px to the <section> tags.
I also implemented a "Back to top" button in the bottom right corner utilizing position: fixed to ensure it remains constantly accessible as the user scrolls.

Images and Styling:
I embedded 5 images throughout the article. I styled them using the global .article-image class to apply uniform rounded corners and shadows. I also utilized customized .callout boxes to visually break up paragraphs and highlight critical environmental facts for the user.

Accessibility

I structured the page strictly using semantic HTML landmarks (<main>, <aside>, <article>) and maintained a proper heading hierarchy (<h1> down to <h2>) to allow screen readers to navigate the document logically.
All 5 images include highly descriptive alt text, and the continuous visibility of the sticky sidebar and "Back to top" button significantly reduces the scroll burden for users relying on keyboard navigation.

Challenges and Lessons Learned

Challenge 1: Anchor Links and Sticky Headers

When implementing the "Back to Top" button on my Content Page, the scrolling behavior glitched. I initially used an id="Top" on my sticky header, which caused case-sensitivity issues with the #top link and fought against the header's sticky positioning.

How I fixed it:
I removed the id from the header completely. I kept the link as <a href="#top">, allowing the browser's native default behavior to scroll to the absolute top of the document without interfering with the sticky CSS.

What I learned:
Modern browsers have built-in reserved keywords (like #top) that natively handle document scrolling. Sometimes, removing custom code and relying on native browser behaviors is cleaner and more reliable than forcing a manual solution.

Challenge 2: Avoiding CSS-Only Hover Effects

The coursework specification strictly required that the Gallery page's hover overlays and expanded views be handled entirely by JavaScript, meaning I could not just use a simple CSS :hover selector for the visual effects.

How I fixed it:
I used querySelectorAll() and a .forEach() loop in JavaScript to attach mouseenter and mouseleave event listeners to every gallery card. These listeners dynamically add and remove an overlay-visible CSS class to trigger the visual transitions.

What I learned:
Strict separation of concerns is highly effective. Using JavaScript to control the behavior (when the class is added) and CSS to control the presentation (how the class animates) keeps the codebase modular and much easier to debug.

Challenge 3: Font Changes Breaking Modal Layouts

While testing the Gallery's user customization panel, I noticed that switching the text to the "Monospace" font caused the letters to take up more horizontal space. This forced the text to wrap to new lines, which vertically stretched the modal card and broke the side-by-side design.

How I fixed it:
I applied a strict height: 550px; limit to the .extended-modal container and added overflow-y: auto; specifically to the right-side content column. This locks the card size in place and simply triggers an internal scrollbar if the text gets too long.

What I learned:
Dynamic user content (like changing fonts or adding text) can easily destroy a rigid design. It is critical to define strict container boundaries and plan for content overflow to maintain structural integrity.
